# Deployment Assist Tool (DAT) — User Manual

Welcome to the **Deployment Assist Tool (DAT)**. This manual provides comprehensive instructions for configuring the required integrations, installing prerequisites, and running DAT either as a local CLI or as a native GitHub App CI/CD orchestrator.

---

## 1. Prerequisites

Before running DAT, ensure your environment has the following dependencies installed:

* **Node.js:** v20+ (`npm` included)
* **Docker:** Required for running OWASP ZAP, Hadolint, and Dockle. Ensure the Docker daemon is running (`docker info`).
* **Python 3:** Required for Garak, Bandit, and pip-audit.
* **Google Cloud SDK (`gcloud` CLI):** Required for deploying the GitHub App and spinning up Ephemeral Preview Environments.
  * *Installation:* [Google Cloud SDK Docs](https://cloud.google.com/sdk/docs/install)
  * *Authentication:* Run `gcloud auth login` and `gcloud config set project YOUR_PROJECT_ID`.

---

## 2. Environment Configuration (`.env`)

DAT relies heavily on external APIs for reporting, LLM remediation, and dynamic scanning. Create a `.env` file in the root directory by copying `.env.example`:

```bash
cp .env.example .env
```

Populate it with your environment-specific configurations:

### GitHub App Configurations
* `APP_ID`: The ID of your registered GitHub App.
* `PRIVATE_KEY`: The RSA private key generated by GitHub (Base64 encoded or multi-line string).
* `WEBHOOK_SECRET`: The shared secret used to verify GitHub webhook payloads.
* `GITHUB_TOKEN`: A Personal Access Token (PAT) used by the Self-Healing CLI Agent to open Pull Requests.

### Google Cloud / Ephemeral Environments
* `GCP_PROJECT_ID`: Your target Google Cloud project ID.
* `GCP_REGION`: The region for Cloud Run deployments (e.g., `us-central1`).
* `GCP_CLOUD_SQL_INSTANCE`: (Optional) Connect ephemeral deployments to a database (`project:region:instance`).

### Tool Integrations
* `GEMINI_API_KEY`: Required for Auto-Distroless LLM Dockerfile refactoring.
* `DEFECTDOJO_URL` & `DEFECTDOJO_API_KEY`: Required if using `--push-dojo`.
* `DEPENDENCY_TRACK_URL` & `DEPENDENCY_TRACK_API_KEY`: Required if using `--push-dtrack`.
* `SONAR_TOKEN`: Required for polling SonarQube API results.

## 3. GitHub App Registration & Setup

To use DAT automatically on Pull Requests, you must configure a Native GitHub App:

1. Navigate to your GitHub Organization Settings -> **GitHub Apps** -> **New GitHub App**.
2. **Name**: Deployment Assist Tool
3. **Webhook URL**: 
   * *Local Dev*: Use a [smee.io](https://smee.io) channel URL.
   * *Production*: The URL of your deployed GCP Cloud Run service (e.g., `https://dat-github-app-xyz.a.run.app/api/github/webhooks`).
4. **Webhook Secret**: Generate a secure random string and paste it here (and in your `.env`).
5. **Permissions Required**:
   * Pull Requests: Read & Write
   * Checks: Read & Write
   * Contents: Read & Write (for self-healing code pushes)
6. **Subscribe to Events**: Check `Pull request`.
7. Generate a **Private Key**, download it, and add it to your `.env` file.
8. Install the App onto your target repositories.

> **Security Note:** DAT enforces an `author_association` check. The pipeline will **only** execute if the PR author is an `OWNER`, `MEMBER`, or `COLLABORATOR`. Anonymous or outside contributors will result in a skipped execution to prevent Denial of Wallet abuse.

## 4. Deploying DAT to Google Cloud Run

Once your `.env` is configured and your GitHub App is registered, you can deploy DAT to GCP to listen for webhooks perpetually.

1. Ensure your `gcloud` CLI is authenticated.
2. Run the deployment script:
   ```bash
   bash infra/gcp/deploy.sh
   ```
3. Once deployed, navigate to the GCP Console -> Cloud Run -> `dat-github-app`.
4. Go to the **Variables & Secrets** tab and securely inject your `.env` variables (e.g., `GEMINI_API_KEY`, `PRIVATE_KEY`).

## 5. Using the DAT CLI Locally

You can run DAT manually against your local workspace to generate Readiness Scores and trigger autonomous fixes before you even open a PR.

### Build the CLI
```bash
npm install
npm run build
```

### Basic Scan
Runs all enabled scanners detected for your ecosystem (Node, Python, Rust, etc.):
```bash
node dist/index.js scan
```

### Advanced Reporting
Generate a PDF report, a SARIF file, and push the SBOM to Dependency Track:
```bash
node dist/index.js scan --pdf results/report.pdf --sarif results/report.sarif --push-dtrack
```

### Targeted Scanning
Skip specific expensive scanners (like ZAP) or only run specific modules:
```bash
node dist/index.js scan --skip zap,k6
# OR
node dist/index.js scan --module security
```

### Ephemeral GCP Deployment for DAST (`--deploy`)
DAST scanners (OWASP ZAP, k6) need a live URL. Instead of supplying one with `--url`, you can
have DAT provision a short-lived **GCP Cloud Run** preview of your workspace, scan it, and tear
it down automatically:
```bash
node dist/index.js scan --deploy --fix-manifest results/dat-fix-manifest.json
```

**Prerequisites:**
* `gcloud` CLI installed and authenticated (`gcloud auth login`).
* A project set via `deployer.gcp.projectId` in `.dat.config.yaml` (defaults to `dat-tool`) or the
  `GCP_PROJECT_ID` env var.
* Docker available (Cloud Build builds the image from `--source .`).
* Cloud Run API + Artifact Registry enabled on the project.

**Cost controls (kept near-zero for test runs):** the preview is deployed with scale-to-zero
(`--min-instances=0`), a single capped instance (`--max-instances=1`), minimal `1` CPU / `512Mi`
memory, **no Cloud SQL** by default, and `--no-allow-unauthenticated` (IAM-only access via a
generated OIDC token). On teardown, DAT deletes **both** the Cloud Run service and its container
image from Artifact Registry so nothing keeps accruing storage cost. Teardown runs even if the
scan fails. Tune CPU/memory/instances and project/region under `deployer.gcp` in the config.

> `--url <target>` always takes precedence: if you pass an explicit URL, provisioning is skipped.
> You can also set `deployer.enabled: true` in the config to deploy on every scan without the flag.

## 6. Understanding the Autonomous Rollback Loop

When DAT detects fixable insecure code patterns (via AST) or Dockerfile vulnerabilities:
1. It applies the fix inline.
2. It looks for your native test framework (e.g., `npm test`, `pytest`, `cargo test`).
3. If tests pass, the fix is kept.
4. If tests fail (or if no test command is found), **DAT will automatically execute `git checkout -- <file>` to revert the changes**, guaranteeing that the pipeline never breaks a functional build.
